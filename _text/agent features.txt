## PART 1: AGENT FEATURES (OpenClaw-Inspired)

### What OpenClaw Is (for context)
OpenClaw is an open-source autonomous AI agent that runs on your machine and connects to messaging apps (WhatsApp, Discord, Telegram) to automate tasks around the clock. It exploded in early 2026, then its creator joined OpenAI. The project moved to a foundation and stays open source.

Key insight: OpenClaw is powerful but deeply technical -- no real UI, no non-developer story. Omnirun's opportunity is to deliver the same agent capabilities wrapped in an interface anyone can use.

### The Core Agent Pattern
All agent features follow the same loop:

```
Agent watches something (email, calendar, GitHub, a file, a website...)
        ↓
Claude processes + summarizes
        ↓
Pushes notification to Omnirun mobile app
        ↓
User replies in plain English
        ↓
Agent takes action (sends reply, updates calendar, creates task...)
```

Build the engine once. Plug in different "watchers."

### Agent Types

**Scheduled agents** -- already exist as Scheduled Tasks. Run on cron.
**Watching agents (new)** -- always-on, event-driven. Triggered by something happening, not by time.

### Future Agent Opportunities

| What it watches       | What it does                                        |
|-----------------------|-----------------------------------------------------|
| Gmail / Outlook       | Summarizes emails, drafts replies, bulk actions     |
| Google Calendar       | Morning brief, conflict warnings, smart scheduling  |
| GitHub PRs / Issues   | "New PR from Sarah -- approve / review?"            |
| Slack / Discord       | Surfaces important messages you missed              |
| Competitor website    | "Their pricing page changed -- here's the diff"     |
| Deployed site         | "Error spike detected -- here's what broke"         |
| A folder / file       | "Client dropped new files -- want me to process?"   |

## PART 2: EMAIL AGENT

### Decision: Do NOT build a full email client
Building a real email client (IMAP/SMTP, thread view, search, attachments, rich compose) would take months and pulls focus from the core product.

### Decision: Build an AI layer on top of Gmail/Outlook via their APIs
The user keeps their existing email client. Omnirun connects via Gmail API (OAuth) or Microsoft Graph API (OAuth) and acts as the AI brain on top.

```
Gmail/Outlook stays as-is
        ↓
Omnirun connects via Gmail API / Microsoft Graph API (OAuth)
        ↓
Omnirun reads, summarizes, drafts, sends -- all via API
        ↓
Interface lives in the Assistant section chat
```

**Example interaction:**
```
Omnirun: "3 emails worth your attention:
  1. Client reply from Marco -- he approved the design
  2. Invoice from hosting provider ($47, due Friday)
  3. Sarah asking for the project files
  Want me to handle any of these?"

User: "Reply to Marco, say great and we'll start Monday.
       Ignore the invoice. Forward Sarah's request to my team."

Omnirun: "Done. Replied to Marco, forwarded to team.
          Reminder set for the invoice Friday morning."
```

**On mobile:** Push notification → tap → opens Assistant chat → handled in one message.

### CRITICAL DISTINCTION: Two Types of Connections

**Project Connections (Settings page)** = project-level services.
Vercel, GitHub, Stripe, Supabase, Netlify, Cloudflare etc.
These are tools the AI uses to build and deploy projects.
They are scoped to projects (global or per-project).
Managed ONLY in Settings → Project Connections.

**Assistant connections** = personal integrations.
Gmail, Outlook, Google Calendar, Slack, Discord etc.
These are the user's personal life/work accounts the assistant monitors on their behalf.
They are NOT project tools.
Managed ONLY in the Assistant section panel.

These two types of connections must NEVER be mixed together in the same settings page.

### Where Email Connection Lives
The Assistant section manages its own connections internally via the left panel (AccountsPanel.tsx).
When no email account is connected, the Assistant section shows a prominent onboarding prompt to connect Gmail or Outlook -- right there in context.
The Home dashboard Connections widget has an "Assistant" tab showing connected accounts as read-only status with "Open Assistant →" link.

### Plan-Gating for Email Accounts
- Starter: 1 email account
- Pro: 2 email accounts
- Business: 5 email accounts
- Enterprise: unlimited

### OAuth Credential Setup
- Option A (implemented): Omnirun registers one Google Cloud app for all users. Credentials stored in .env (VITE_GMAIL_CLIENT_ID, VITE_GMAIL_CLIENT_SECRET). Users just click Connect.
- Google OAuth verification required before public launch (4-6 weeks review). Can test with up to 100 manually added test users during development.
- Same pattern for Microsoft Azure / Outlook.

### OAuth Flow (FULLY BUILT -- April 11, 2026)
1. User clicks "Connect Gmail" in AccountsPanel (or "+" button next to provider)
2. ConnectAccountModal opens -- if credentials in .env, skips to OAuth; if not, shows configure step
3. Tauri invokes start_gmail_oauth command (oauth.rs -- BUILT for all 9 providers)
4. Rust spawns localhost HTTP server on port 49580, opens system browser with OAuth URL
5. User logs in, provider redirects to localhost/callback (or Cloudflare Worker for Slack/Notion)
6. Rust exchanges auth code for tokens, fetches user info, returns OAuthResult to frontend
7. Frontend saves to Supabase assistant_email_accounts + local SQLite cache
8. Account appears in panel, modal shows success step

### HTTPS Redirect Worker (for Slack/Notion)
- Slack and Notion require HTTPS redirect URIs -- localhost HTTP is rejected
- Cloudflare Worker deployed at https://oauth-redirect.omnirun-app.workers.dev
- Worker redirects to http://127.0.0.1:49580 with query params preserved
- oauth.rs uses this URL for slack/notion providers, localhost for all others

### Gmail Email Service (BUILT -- April 11, 2026)
- emailService.ts: full Gmail API service via gmail_api_proxy Rust command
- Read: fetchRecentEmails, fetchUnreadEmails, searchEmails, getEmailById
- Send: sendEmail (RFC 2822 format, supports replies with threading)
- Auto-refresh: tokens refresh silently using refresh_token + stored OAuth credentials
- AI integration: AssistantChatArea detects email queries, prefetches data, injects into system prompt
- Send flow: AI drafts email with To/Subject/Body, user confirms with "Send it", parseDraftAndSend executes

- Google Calendar: CONNECTED (API enabled, OAuth works, no calendar service built yet)
- Discord: CONNECTED (OAuth works, no message service built yet)

## PART 5: MEMORY SYSTEM -- BUILT (Layers 1+2) + IMPROVED (April 2, 2026)

See memory-system.txt for the full implementation guide.

### Philosophy
Remember the way a good human collaborator remembers. Not verbatim logs, but meaning. Not everything, but what matters. Not static facts, but evolving understanding.

### Guiding Principles (added April 2, 2026)
- **Bandwidth-aware**: Every token has a cost. Prune or compress anything low-value, stale, or derivable.
- **Skeptical memory**: Recalled info treated as hints. Agent verifies architecture claims against actual files before acting.
- **Strict write discipline**: Memory updates gated on confirmed success. Failed attempts stored as failures, not facts.
- **Derivability rule**: If a fact can be re-derived from config files / codebase, don't store it. Save budget for preferences, decisions, failures.
- **Self-healing**: Memory improves itself during idle time, not just weekly.

### Three-Layer Architecture

**Layer 1: user_context.md (global identity file)**
- Always loaded into every Assistant AND Project chat session
- Maximum 3000 tokens (hard cap, enforced by multi-phase compression)
- Stored in SQLite user_memory table
- User can view and edit via "About me" panel in Assistant section
- Updated by compression job + idle consolidation

**Layer 2: project files (.omnirun/context.md)**
- One file per project, loaded only when working on that project
- NOT created on disk until AI has real content to save (empty guard in saveContext)
- Stores NON-DERIVABLE knowledge only: decisions + WHY, failed approaches, naming conventions, deployment quirks
- Tech stack / file structure NOT stored (derivable from package.json / file reads)
- Updated after each session that touches that project, gated on confirmed success

**Layer 3: vector store (sqlite-vec)**
- Local SQLite extension -- no external database needed, fits privacy model
- Contains compressed session summaries, specific decisions, things tried that didn't work
- Queried at session start: retrieves top 3-5 chunks relevant to current context
- Grows indefinitely without bloating context window
- Requires sqlite-vec Rust crate addition to Cargo.toml

### User Controls (Settings → Memory section)
Three toggles, all default ON:
- "Allow Omnirun to learn from your Assistant conversations — the more you use it, the better it understands your habits, preferences and style. Everything is stored locally on your device and never leaves it."
- "Allow Omnirun to learn from your Project conversations — stores coding preferences, architectural choices, your naming conventions. Stored locally only."
- "Allow Omnirun to learn your writing voice — used when drafting emails and content on your behalf. Stored locally only."

Sync toggle (default OFF):
- "Sync memory across devices — your memory will be encrypted and stored in the cloud so it works across all your installations. Note: in sync mode, your memory is no longer stored locally only."
- Confirmation: "Turning on sync will upload your memory to Omnirun's secure servers. It is encrypted before leaving your device, but it will no longer be local-only. [Cancel] [Sync]"

### Observation Extraction
- Runs after conversations with 4+ messages exchanged
- Uses Haiku (cheap, fast) -- completely silent background call
- Extracts: approvals/rejections, explicit preferences, recurring people, behavioral patterns, voice/style signals
- Never stores: raw email content, credentials, third-party contact details (first name only), raw code
- Derivability check: skips facts readable from config files / codebase
- Success gate: marks failed operations as FAILED_ATTEMPT (prevents memory pollution)

### Multi-Phase Compression (improved April 2, 2026)
- Weekly (Sunday) via Scheduled Tasks system + idle consolidation (30+ min idle, 10+ observations, 4h cooldown)
- Also triggered when raw observations hit 50 entries
- 4-phase structured prompt (single Haiku call, same cost, better output):
  1. Deduplicate -- skip observations already captured
  2. Contradiction check -- update only if contradicted 3+ times
  3. Staleness check -- remove derivable facts, demote entries not reinforced in 60+ days
  4. Compress & write -- merge, sharpen, enforce 3000 token cap
- Hard 3000 token cap -- recency + frequency determine what stays

### Verification Nudge (added April 2, 2026)
- ~50 tokens in system prompt distinguishing trusted vs hint memory
- Preferences, patterns, past failures = TRUSTED (act on directly)
- Architecture, tech stack, dependencies, file structure = HINTS (verify against actual files)
- Prevents stale-memory-induced hallucinations when project state has changed

### Idle-Time Consolidation (added April 2, 2026)
- runIdleConsolidation() in memoryService.ts
- Triggers: 30+ min idle, 10+ unprocessed observations, 4+ hours since last run
- Reuses multi-phase compression, costs ~$0.001-0.003 per run
- Keeps memory fresher than weekly-only compression

### Onboarding Step
- New step after existing onboarding (before CompletionStep)
- Explains memory in plain English, shows 3 toggles, defaults all ON
- "You can change this anytime in Settings"
- Cold start: asks 3-4 natural questions in first conversation (what are you building, how technical are you, what should I handle automatically vs always ask about)

### Privacy Boundaries
- Never stored: raw email content (summaries and patterns only), credentials (regex filter), third-party email addresses (first name only), file contents (patterns and decisions only, not code)
- "Memory updated" indicator after extraction -- user can tap to see what was added
- Explicit forget: "Forget that I mentioned X" works immediately
