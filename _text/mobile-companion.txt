## What Is It?

A native mobile app (App Store + Play Store) that connects to the user's desktop Omnirun app through Supabase. It's a remote control --  lets users monitor, control, and trigger actions on their computer remotely.

The core idea: **your desktop is the brain, your phone is the remote.** The mobile app doesn't run AI, doesn't process files, doesn't build anything. It mirrors what's on your desktop and lets you control it from anywhere.

NOT a full version of Omnirun on mobile. It's a remote control + dashboard.

---

## Tech Stack

- **React Native** (shares React knowledge from desktop app)
- **Supabase Realtime** for desktop   mobile communication
- **Supabase Edge Functions** for Wake-on-LAN (sends magic packet via UDP)
- **Firebase Cloud Messaging** (Android) + **APNs** (iOS) for push notifications

## Features --  Grouped by Priority

### Priority 1: Remote Power & Status

**Wake-on-LAN (Turn Computer On)**
- Desktop app auto-detects MAC address, local IP, and public IP on first setup
- Desktop app auto-enables WoL on network adapter (OS-level command)
- Desktop app attempts UPnP auto port forwarding on router
- Two wake methods (user picks in Settings > Remote Access):
  - WoL Local: magic packet on same WiFi network (also wakes from sleep)
  - WoL Port Forward: magic packet to public IP via forwarded UDP port (works from anywhere)
- One-button setup: user clicks "Enable Remote Wake" and app auto-configures everything
- Fallback guide only if UPnP fails (manual port forwarding) or WoL needs BIOS change
- No extra devices or purchases required
- Zero credentials stored in Supabase -- only MAC address, IPs, port number
- BIOS "Power On After AC Loss" recommended for auto-boot after power outage

**Remote Shutdown / Restart / Sleep**
- Mobile sends command   Supabase   desktop app executes OS command
- Confirmation prompt on mobile ("Are you sure you want to shut down?")
- Desktop app listens via Supabase Realtime channel for remote commands

**Computer Status**
- Online/offline indicator (desktop app sends heartbeat every 60 seconds to Supabase)
- Last seen timestamp if offline
- Basic system info: OS, uptime

### Priority 2: Chat Mirroring (Remote Chat Control)

**This is the killer feature.** The user controls the full desktop Omnirun chat from their phone. The desktop does all the work --  AI calls, file operations, tool execution --  the mobile app is just a remote screen with an input field.

**How it works:**
```
User types on mobile
message sent via Supabase Realtime
Desktop app receives message
Injects into chat (same as if user typed it at the keyboard)
AI processes with full project context + file access + tools
AI response streams back
Desktop syncs response to Supabase Realtime
Mobile displays response in real-time
```

**What the user sees on mobile:**
- Current project name at top
- Full chat conversation (synced from desktop)
- Text input at bottom
- AI responses appear in real-time as they stream on desktop
- Tool execution status visible ("Writing file...", "Reading project...")

**What this enables:**
- User on the couch tells AI to build a page --  AI creates files on the computer
- User in bed asks AI to fix a bug --  AI edits code on the computer
- User at a caf triggers a full project build --  computer does all the work
- Same quality as sitting at the desk, because it IS the desktop running everything
- **(v2) Desktop app control from mobile:** User types "open Figma and create a new frame" on phone -- desktop takes screenshots, AI controls the app, user sees progress in mobile chat. See desktop-app-control.txt for full spec.

**What the mobile does NOT do:**
- No local AI processing
- No local file access
- No tool execution
- It just sends text and receives text + status updates

**Project switching from mobile:**
- User can see project list and switch active project
- Desktop switches to that project (loads context, file tree, etc.)
- Chat history for that project loads on mobile

**Image support (multi-device):**
- User takes a photo on phone   sends via Supabase   desktop receives + sends to AI as vision input
- Great for: "Here's a sketch of the layout I want" or "Here's an error on my phone screen"

**Key technical details:**
- Messages sync via Supabase Realtime channel: `chat:{user_id}:{project_id}`
- Desktop sends: assistant messages, tool status updates, streaming chunks
- Mobile sends: user messages, project switch commands
- Conflict handling: if someone types on both desktop and mobile simultaneously, desktop input takes priority (simple queue)
- Chat history stored on desktop (SQLite), mobile just mirrors current session
- If desktop is offline, mobile shows "Computer is offline" and offers to wake it

## Architecture: How Desktop   Mobile Communicate

```
Mobile App   Supabase (Realtime + Database)   Desktop App

Communication channels (Supabase Realtime):
- device:{user_id}:status    heartbeat, online/offline, CPU/RAM/disk
- device:{user_id}:commands    mobile sends commands, desktop listens
- device:{user_id}:responses    desktop sends results back
- chat:{user_id}:{project_id}    chat mirroring (messages both ways, streaming)
- tasks:{user_id}:updates    task status changes, push to mobile
```

**Desktop app responsibilities:**
- Sends heartbeat every 60 seconds (I'm alive, here's CPU/RAM/disk)
- Listens on commands channel for remote instructions
- Listens on chat channel for mobile messages   injects into chat
- Streams AI responses back to chat channel in real-time
- Reports task execution results
- Syncs project list and status periodically

**Mobile app responsibilities:**
- Shows dashboard from Supabase data
- Sends commands (wake, shutdown, run task, deploy)
- Sends chat messages   desktop processes them
- Displays chat responses streamed from desktop
- Receives push notifications for failures/alerts

**Supabase responsibilities:**
- Stores device status, task states, project metadata
- Relays real-time messages between mobile and desktop
- Edge Functions handle Wake-on-LAN packet sending
- Row-level security: users only see their own data

## Wake-on-LAN --  Technical Detail

**Setup flow (one-time, in desktop app):**
1. Desktop app detects MAC address and local IP automatically
2. Saves to Supabase (encrypted)
3. User enables WoL in BIOS (desktop app shows guide with screenshots per manufacturer)
4. Test: user walks away, tries to wake from mobile
5. If test fails: troubleshooting guide (router settings, BIOS settings)

**How the wake works:**
```
Mobile   HTTPS request   Supabase Edge Function   UDP magic packet   Computer wakes up
```

**Local network wake:** Easy, works reliably. Edge Function sends packet to the local network broadcast address.

**Internet wake (computer on different network):** Desktop app attempts UPnP auto port forwarding during setup. If UPnP works, it's automatic. If not, user follows a one-time manual port forwarding guide. No extra devices or purchases needed.

## Power Recovery (Computer Was Off Due to Power Outage, etc.)

When a computer loses power completely (outage, unplugged, etc.), no software can turn it on --  there's no electricity. But once power is restored, the user has two options. Both are presented during onboarding, user picks their preference.

**Option A: Manual Wake from Phone (Wake-on-LAN)**
- Power comes back   computer stays off but has electricity   network card listens
- User opens mobile app   sees "Computer offline"   taps "Wake Up"
- WoL magic packet sent   computer boots   Omnirun auto-starts   connects to Supabase
- Mobile shows "Computer is back online"
- Best for: users who want full control over when their computer is on

**Option B: Automatic Power-On (BIOS setting, recommended)**
- User enables "Power On After AC Loss" / "Restore on AC Power Loss" in BIOS (one-time setup)
- Power comes back   computer boots automatically   Omnirun auto-starts   connects to Supabase
- User gets push notification: "Your computer is back online"
- No action needed from the user at all
- Best for: users who travel, want always-on reliability, or run scheduled tasks

**What we build:**
- Omnirun desktop app set to auto-start with OS (so it's always running after boot)
- BIOS setup guide in onboarding for both WoL and auto-power-on (with screenshots per manufacturer)
- Mobile app shows "Computer offline since [timestamp]" so user knows something happened
- Push notification when computer comes back online
- Any scheduled tasks or queued commands that were missed during downtime auto-run on reconnect
- User picks Option A or B during setup (can change anytime in Settings)

## Pairing Flow (Connect Mobile to Desktop)

**V1: Same login = auto-connected (simplest)**
1. User logs into desktop Omnirun with their account
2. User downloads mobile app, logs in with same account
3. Both apps connect to the same Supabase channels automatically
4. Mobile sees the computer, projects, tasks, chat. Done.

No QR codes, no PINs, no setup. Just login on both devices.

**Multi-device (only needed if user has multiple computers)**
- Desktop shows list of paired computers in Settings   Devices
- Mobile lets user pick which computer to control
- QR code or 6-digit code pairing for adding new computers

## Key Decisions

- **Chat mirroring over separate mobile AI** --  desktop does all the work, mobile is just a remote. Simpler to build, same quality, zero extra AI cost.
- **React Native over Flutter** --  reuses React knowledge, shared component logic concepts with desktop app
- **Supabase as the bridge** --  already needed for desktop auth/teams, no extra service
- **No full app building on mobile** --  mobile is remote control, not a replacement for desktop
- **Security first** --  all commands require confirmed pairing, remote terminal has same blocklist as desktop, Wake-on-LAN credentials encrypted
