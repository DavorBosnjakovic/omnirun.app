1. Core Philosophy: Solving Context Entropy in Production Agent Systems
Long-running coding agent sessions suffer from “context entropy”: the prompt becomes bloated with stale tool outputs, repetitive chat turns, contradictory facts, superseded decisions, verbose logs, or low-value noise. Consequences include hallucinations, inconsistent reasoning, token waste, degraded performance, and poor resumption across sessions/restarts.
Claude Code treats context and memory as a first-class, bandwidth-aware, skeptical, and self-healing system — not an afterthought or simple “bigger window” fix. Key guiding principles repeatedly highlighted in analyses:

Bandwidth-aware: Every token has a cost. Aggressively prioritize high-signal information; prune or compress anything low-value, stale, or derivable from the current codebase/state.
Skeptical / hint-based memory: Recalled information is always treated as hints or pointers, never absolute truth. The agent must verify against ground truth (files, codebase, logs) before acting.
Strict Write Discipline: Memory index or topic-file updates are gated — they only occur after a confirmed successful operation (e.g., file write succeeded). This prevents pollution from failed attempts, hallucinations, or unverified observations.
Self-healing & autonomous: Memory actively improves itself in the background, merging observations, deduplicating, resolving contradictions, pruning stale data, and converting vague insights into durable, structured facts.
Automatic & low-interruption: Most maintenance runs without user input, enabling reliable multi-hour, multi-day, or resumed workflows.
Derivability rule: If a fact can be re-derived from the current codebase or state (via tools like grep, LSP, or file reads), it is often not redundantly stored.
Hierarchical & scoped: Memory operates at global/user, project, and session levels.

The system explicitly separates active context (what’s in the prompt right now) from persistent memory (what survives and self-improves across sessions).
2. Context Window Management (Active / In-Prompt Context Orchestration)
A dedicated orchestration layer continuously estimates token usage across system instructions, memory index, conversation history, tool results, and current task state. It uses prompt-caching awareness to minimize invalidation.
Triggers for compaction/intervention:

Proactive soft thresholds (commonly ~70–80% of model limit, sometimes higher).
Reactive signals (prompt-too-long errors, repetition, contradictions, or entropy indicators).
Event-driven (after heavy tool use, long turns, scheduled checks, or session milestones).

Five Cascading Compaction Strategies (applied hierarchically or in combination, chosen dynamically by context type and pressure):

Time-based / staleness clearing of tool results — Drop oldest or least-relevant outputs first (they become obsolete fastest).
Recent conversation summarization — Condense the latest N turns into a compact, high-level summary while preserving key decisions/state.
Session memory extraction — Pull important facts, project state, decisions, or learnings into persistent layers.
Full history / thread summarization — Broader compression when deeper reduction is needed.
Oldest-message truncation — Last-resort fallback to remove earliest history.

Supporting mechanisms:

Detailed token breakdown and visualization tools (showing spend on memory index vs. history vs. tools).
Structured “compact boundary” markers or continuity-preserving elements.
Aggressive deduplication of file reads and tool result sampling.
Integration with session state: automatic extraction of task specs, file lists, errors, workflow progress, and recent commits/git branch info.
Cascading fallback logic for graceful degradation.

This keeps the active context lean, focused, and high-signal even in very long or resumed sessions.

4. Memory Extraction, Consolidation & Self-Healing (autoDream / Dream Mode)
Memory is dynamic and runs autonomous improvement processes.
Extraction:

Proactive after key events: task completion, user corrections, successful tool uses, pattern detection, or session milestones.
Scans recent activity to identify useful observations, rules, decisions, or insights.
Converts implicit/vague information into concise, structured, verifiable entries in the appropriate layer.

Consolidation Cycle (autoDream / Dream Mode):
Runs as a forked sub-agent (leveraging prompt caching for near-zero extra cost) during idle periods, between sessions, or via background processes. The cycle typically follows four phases:

Orient / Read — Scan the index and relevant topic files to establish the current knowledge baseline.
Gather / Collect — Review recent logs/transcripts (narrowly, without full reloads) for new observations, corrections, decisions, contradictions, or learnings.
Consolidate / Merge — Deduplicate entries, resolve conflicts, sharpen fuzzy insights into clear facts, merge related items, standardize details.
Prune & Store — Update the lightweight index, reorganize or split topic files, remove stale/outdated/low-value items, enforce size caps, and rebuild references. Often produces a brief summary of changes.

KAIROS (Autonomous Background Daemon Mode):

An always-on / persistent capability (referenced extensively in analyses).
Allows the agent to operate autonomously while the user is idle.
Receives periodic “tick” prompts to decide whether to act proactively.
Includes autoDream for memory consolidation, merging observations, removing contradictions, and turning vague insights into verified facts.
Maintains append-only daily log files and can subscribe to external events (e.g., GitHub webhooks in some descriptions).
Runs with safety budgets, guardrails, and read-only access where needed to prevent disruption or excessive resource use.
Enables true persistence, self-maintenance, and light background tasks across sessions without constant user prompting.

This “REM sleep”-like self-healing loop is credited with maintaining long-term coherence and reliability in multi-day projects.
5. Session Persistence, Resumption & Related State Handling

Full activity is logged persistently.
On resume (via flags, session IDs, or continue commands), the system automatically extracts and injects relevant state: task specifications, file lists, recent decisions/errors, workflow progress, and key learnings.
Session-specific structured memory folds into longer-term layers during or after the session.
Compaction integrates with resumption to preserve continuity via markers or extracted summaries.

6. Broader Supporting Patterns & Features

Multi-agent / sub-agent coordination: Fork model (efficient byte-identical context copies via prompt caching), teammate/mailbox model, worktree/isolation model. Shared team/project memory where applicable.
Token/cost awareness and optimizations throughout (including repo state like recent commits/branch info).
Repo integration: Selective inclusion of project state without bloat; custom grep/glob/LSP usage; file-read deduplication.
Safety and discipline: Retries, permission schemas, sandboxing, boundaries, and error-handling to prevent runaway growth or repeated failures.
User influence: Persistent instruction/guidance files read automatically and can shape memory behavior.
Background autonomy: KAIROS enables observation, logging, and light proactive actions with built-in guardrails.
