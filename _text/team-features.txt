# Omnirun --  Team Features Specification

**Status:** Built
**Last Updated:** April 6, 2026

---

## OVERVIEW

Teams allow Pro, Business, and Enterprise users to collaborate on projects.
The system is intentionally simple --  no real-time collaborative editing,
no file-level locking, no merge conflicts. One person works on a project
at a time. The AI needs full control of the project folder, so simultaneous
editing would cause chaos.

---

## TERMINOLOGY

- **Team** --  A group of users under one paid plan
- **Members** --  People in the team
- **Owner** --  The person who pays. One per team. Cannot be changed (only transferred).
- **Member** --  Everyone else on the team. Can access projects, chat with AI, deploy.

Only two roles: **Owner** and **Member**. No Admin/Editor/Viewer.
Granular permissions not included  --  just Owner + Member.

## INVITATION FLOW

1. Owner clicks [+ Invite Member]
2. Modal appears: "Enter email address" + [Send Invite]
3. System sends email to invitee:
  - "You've been invited to join [Team Name] on Omnirun"
  - Two buttons: "Download Omnirun" (if new) + "Accept Invitation"
  - Invitation expires after 7 days
4. Invitee downloads app (if needed), creates account, accepts invite
5. They appear in the member list
6. Owner sees "Invitation pending" status until accepted

### Invitation States
- **Pending** --  Sent, not accepted yet. Shows in member list as gray/italic.
- **Accepted** --  Active member.
- **Expired** --  7 days passed. Owner can resend.

### Limits
- Cannot invite more members than available seats
- Cannot invite someone already on another team (one team per account)
- Owner can cancel pending invitations

## API KEY POLICY

Owner chooses one of two modes in Team Settings:

### Shared Key (Default)
- Owner's API keys are encrypted (AES-GCM via Web Crypto API) and stored in Supabase teams.encrypted_api_keys
- Encryption key derived from team_id + owner_id -- Supabase RLS is the real security layer
- Members don't need to set up API keys at all
- All AI costs go to the owner's API account
- Best for: Agencies where the boss pays for everything
- Members see "Using team API key" banner in API Settings (entire page replaced, read-only)
- Owner sees "Your API keys are shared with your team" notice in API Settings
- When owner saves API key changes, keys are re-encrypted and re-uploaded automatically

### Individual Keys (BYOK Per Member)
- Each member enters their own API key in their personal API Settings
- Each member pays their own AI provider directly
- Best for: Loose teams, freelancer groups
- If a member hasn't set up their key, they see info banner: "Each team member uses their own API key. Set yours up in Settings → API Key."

The toggle lives in Team Settings, only the Owner can change it.

### Confirmation Dialogs

**Switching to Shared Key:**
"Share your API key with your team?"
"Your API keys will be encrypted and stored securely on Omnirun's servers so your team members can use them. They will no longer be stored only on your device. All AI costs will go to your API account. You can switch back to individual keys at any time."
Buttons: [Cancel] [Share Key]

**Switching to Individual Keys:**
"Stop sharing your API key?"
"Your API key will be removed from Omnirun's servers. Team members will need to set up their own API keys to keep working."
Buttons: [Cancel] [Stop Sharing]

### Implementation Details
- Supabase migration 005: added encrypted_api_keys (TEXT) and shared_keys_updated_at (TIMESTAMPTZ) to teams table
- Encryption: AES-GCM, key derived from SHA-256(team_id:owner_id:omnirun-team-key), IV prepended to ciphertext, base64 encoded
- All provider configs (providerId, apiKey, selectedModel) are encrypted as one JSON blob
- Existing RLS policies cover access: "Members read own team" + "Owners update own team"
- teamStore.ts handles encrypt/decrypt via setApiKeyPolicy() and fetchSharedKeys()

## PROJECT-LEVEL LOCKING

Since AI modifies files across the whole project, only one person can work
on a project at a time. This is the core collaboration rule.

### How It Works

1. Member A opens Project X and starts chatting with AI
2. Project X is now **locked** to Member A
3. Other members see Project X in their sidebar with:
  - Grayed out appearance
  - Lock indicator + message: "Sarah is working on this"
  - Project is read-only (can browse files but cannot chat/modify)
4. Member A closes the project / switches to another project / goes offline
5. Project X **unlocks** automatically
6. Other members get notified: "E-commerce Site is now available"

### Lock Behavior
- Lock is automatic --  no manual "lock/unlock" button
- Lock activates when a member opens a project AND sends their first message
- Just browsing files doesn't lock (only active AI work does)
- Lock releases when: member switches project, closes app, or goes idle 15+ min
- If app crashes, lock auto-releases after 5 minutes of no heartbeat
- Owner can force-unlock any project from Team Settings (emergency override)

### UI Indicators
- **Sidebar project list:** Small avatar/dot next to project name showing who's in it
- **Locked project:** Gray text, lock icon, "Sarah is working on this" tooltip
- **Available project:** Normal appearance, green dot = no one working on it

## NOTIFICATIONS

Minimal. Two types only:

### In-App Notifications
- **Project available:** "E-commerce Site is now available" (when someone finishes)
- **Invitation received:** "You've been invited to join [Team Name]"

### How They Appear
- Small dot indicator on the bell icon in sidebar (or topbar)
- Click to see notification list
- Notifications auto-dismiss after being read
- No email notifications
- No sounds, no toast popups, no push notifications

---

## MEMBER REMOVAL

- Owner clicks [x] next to a member's name
- Confirmation: "Remove Sarah Miller from the team? She'll lose access immediately."
- On confirm: Access revoked instantly, seat freed up
- Nothing is deleted --  all projects and files remain intact
- Removed member's app reverts to logged-out / personal state
- Activity log records: "Sarah Miller was removed --  Feb 10"

---

## ACTIVITY LOG

Lives in Team Settings page. Simple chronological list.

### What Gets Logged
- Member worked on [Project Name] --  timestamp
- Member deployed [Project Name] --  timestamp
- Member joined the team --  timestamp
- Member was removed --  timestamp
- Project created --  timestamp
- Project deleted --  timestamp

### Behavior
- Shows last 30 days
- Refreshes when Team Settings page is opened (NOT real-time)
- Grouped by day ("Today", "Yesterday", "Feb 8", etc.)
- No filtering or search

---

## BILLING --  OWNER ONLY

- Only the Owner sees the Billing tab in Settings
- Members do NOT see: plan info, pricing, payment method, billing history
- Members see Team tab but with read-only info (member list, activity)
- If a member clicks Billing in sidebar, redirect to Team page or hide it entirely

---

## NOT INCLUDED

These are not being built. Do NOT build:
- [ ] Granular roles (Admin/Editor/Viewer) --  just Owner + Member
- [ ] File-level locking or collaborative editing
- [ ] Real-time sync via WebSockets
- [ ] Conflict resolution / merge logic
- [ ] In-line code comments between team members
- [ ] Team chat / messaging
- [ ] Project-level permissions (all members see all team projects)
- [ ] Multiple teams per account
- [ ] Transfer ownership UI (manual/support only)
- [ ] Email notifications
- [ ] Usage analytics per member

These can be added later based on actual user demand.

---

## BACKEND REQUIREMENTS

Team features need a backend. Minimum infrastructure:

- **User accounts / auth** --  Email + password, or magic link
- **Team database** --  Teams table, Members table, Invitations table
- **Project lock service** --  Simple flag per project (locked_by, locked_at)
- **Heartbeat** --  Periodic ping from app to confirm member is still active (for auto-unlock)
- **Activity log storage** --  Append-only log per team
- **Email service** --  For sending invitations (SendGrid, Resend, etc.)

This overlaps with Backend/Services in DEVtasks (Stripe, subscriptions, license validation).
Build together --  don't create two separate backend systems.
