# Token Optimization Checklist

Based on analysis of aiService.ts and toolService.ts (Feb 7, 2026)
Updated: Feb 11, 2026 -- Phase 1 + Phase 1.6 + Phase 2 + Phase 3.1 + Phase 3.2 + Phase 4.1/4.2 + XML parser fix + XML entity unescaping done

## Current Problem (ORIGINAL BASELINE)
- 40.2k input tokens vs 5.1k output -- input is 6x the problem
- System prompt sends full project manifest + open file content every call
- Connection prompt sends all 8 providers even when none connected
- Tools prompt has oversized HTML examples
- Chat history grows unbounded with full tool results (file contents stay forever)

## Real-World Results (Feb 8-10 2026)
- DeepSeek Reasoner -- simple task (fresh): $0.074 / 5 calls / 16.2k tokens (3.2k avg/call)
- DeepSeek Reasoner -- follow-up: $0.204 / 14 calls / 49.8k tokens (history bloat)
- DeepSeek Reasoner -- after XML fix: $0.470 / 35 calls / 112.7k tokens (model loops, not code issue)
- DeepSeek Chat -- same task: $0.012 / 3 calls / 3.5k tokens <-- THIS IS THE TARGET
- DeepSeek Chat -- after Phase 3.1+3.2: last call 2.4k tokens / <$0.01 (confirmed Feb 9)
- edit_file confirmed working: "Edited: styles.css (replaced 30 chars with 41 chars)"
- Main remaining problem: DeepSeek Reasoner is terrible at tool use (35 calls for a color change)
- DeepSeek Chat is excellent: 3 calls, $0.012, done correctly

### Key Finding
DeepSeek Reasoner is a thinking/reasoning model, NOT suited for agentic tool use.
DeepSeek Chat (V3) is the right model -- fast, cheap, efficient with tools.
Model choice matters more than any single optimization.

## Compatibility Fix: Multi-Format Tool Call Parser -- DONE

### Problem
DeepSeek models use different tool call formats than Claude/OpenAI:
- Claude/OpenAI: `<tool_call>{"name": "read_file", "arguments": {"path": "file.css"}}</tool_call>`
- DeepSeek variant 1: `<read_file><path>styles.css</path></read_file>`
- DeepSeek variant 2: `<tool_call><n>read_file</n><arguments><path>file.css</path></arguments></tool_call>`

All three caused parse failures -> AI loops endlessly -> burns tokens for nothing.

### Fix (toolService.ts)
- [x] Added XML_TOOL_NAME_MAP -- unified alias map for all tool names (canonical + 20+ aliases)
- [x] Strategy 1 (existing): `<tool_call>{JSON}</tool_call>` -- Claude/OpenAI format
- [x] Strategy 2 (new): `<read_file><path>...</path></read_file>` -- bare XML format
- [x] Strategy 3 (new): `<tool_call><n>read_file</n><arguments>...</arguments></tool_call>` -- XML inside tool_call
- [x] Strategy 4 (existing): Line-based fallback
- [x] parseXmlStyleToolCalls() -- handles bare XML tool tags
- [x] parseXmlToolArgs() -- extracts params from XML child tags, trims whitespace, **unescapes XML entities**
- [x] parseXmlInsideToolCall() -- handles XML structured content inside <tool_call> wrapper
- [x] Claude format unaffected -- hits Strategy 1 first, never reaches new code
- **Result:** DeepSeek Chat successfully completes tasks (3 calls, $0.012)
- **Files modified:** toolService.ts only

## Compatibility Fix: XML Entity Unescaping -- DONE (Feb 9, 2026)

### Problem
When AI writes HTML inside XML tool calls (e.g. `<content>&lt;!DOCTYPE html&gt;...</content>`),
the HTML entities were written to disk as-is. Files contained literal `&lt;` instead of `<`.
This caused the static preview to show raw escaped HTML source instead of rendering the page.

### Fix (toolService.ts)
- [x] Added `unescapeXmlEntities()` function -- &lt; -> <, &gt; -> >, &amp; -> &, &quot; -> ", &apos; -> '
- [x] Applied in `parseXmlToolArgs()` -- values get unescaped at parse time
- [x] Safety net in `sanitizeFileContent()` -- catches escaped HTML before writing HTML files
- **Result:** AI-generated static HTML websites now render correctly in preview

## Phase 6: Desktop App Control Token Optimization (v2)

### See desktop-app-control.txt for full cost breakdown

### The Problem
Desktop app control uses Claude Vision to read screenshots. Each screenshot costs ~1,200-1,600 input tokens. A typical task takes 5-7 screenshots = ~17,500 tokens. At Sonnet pricing that's ~$0.05/task. At 20 tasks/day = ~$30/month added to user's bill.

### Optimization Strategies

#### 6.1 Default to Haiku for screen reading
- [ ] Route all screen control calls to Haiku 4.5 ($0.80/1M tokens)
- [ ] Haiku is surprisingly good at reading screenshots and outputting coordinates
- [ ] Only escalate to Sonnet when Haiku fails or returns low confidence
- **Saves:** ~75% vs Sonnet ($0.014/task instead of $0.053/task)

#### 6.2 Crop to active window
- [ ] Screenshot only the active window, not the full screen
- [ ] Use OS APIs to get window dimensions (already needed for active window detection)
- [ ] Half the pixels = roughly half the vision tokens
- **Saves:** ~30-40% on screenshot tokens

#### 6.3 Resolution downscaling
- [ ] Scale screenshots to 1280x720 or 960x540 before sending
- [ ] Most UI elements are readable at lower resolutions
- [ ] User-configurable: Low (540p) / Medium (720p) / High (1080p)
- **Saves:** ~20-40% depending on original resolution

#### 6.4 Action batching
- [ ] Let AI plan 2-3 actions from one screenshot when confident
- [ ] "Click width field, type 1440, Tab, type 900" = 1 screenshot instead of 3-4
- [ ] AI decides when batching is safe vs when it needs to verify
- **Saves:** ~30-50% fewer screenshots per task

#### 6.5 Skip confirmation screenshots
- [ ] Don't screenshot after every action -- only when AI needs to navigate or verify
- [ ] Trust that typing into a focused field worked (screenshot only if unsure)
- **Saves:** ~20-30% fewer screenshots per task

#### 6.6 Cache screen control system prompt
- [ ] The ~1,000 token instruction set is identical every call
- [ ] Already covered by Phase 1.6 prompt caching (Anthropic only)
- [ ] For non-Anthropic providers: include in static portion of system prompt
- **Saves:** 90% off system prompt tokens (already implemented, just applies here too)
