## 3. FILE CONTENT SANITIZATION   IMPORTANT     DONE

### What was done
- Added sanitizeFileContent() function
- Cleans escaped quotes (\"   ") and escaped slashes (\/   /) in HTML/HTM files only
- Called in write_file case before writeFile()
- Also called in create_directory auto-correct write path

### XML Entity Unescaping Fix (Feb 9, 2026)
**Root cause found:** When AI models (especially DeepSeek) use XML-style tool calls like
`<write_file><path>index.html</path><content>&lt;!DOCTYPE html&gt;...</content></write_file>`,
the HTML content arrives with XML entities (&lt; &gt; &amp; etc.) that must be decoded.

**Fix applied in toolService.ts:**
1. Added `unescapeXmlEntities()` helper   converts &lt;   <, &gt;   >, &amp;   &, &quot;   ", &apos;   '
2. Applied in `parseXmlToolArgs()`   all values extracted from XML tags get unescaped at parse time
3. Added as safety net in `sanitizeFileContent()`   catches any escaped HTML before writing to disk
4. This was the actual cause of the "preview shows raw HTML source" bug   files contained literal &lt; on disk

### Checklist
- [x] Add sanitizeFileContent() function

- [x] Also call it in the create_directory auto-correct write path

## 6. DELETE CONFIRMATION   EVALUATED   INTENTIONALLY SKIPPED

### Decision (Feb 9, 2026)
Evaluated both options (block directory deletion vs confirmation UI).
Implemented Option A (block directory deletion), then reverted after testing.

### Reasoning
- Blocking folder deletion is unnecessary   if user asks to delete a folder, they want it deleted
- Path traversal protection (Section 1) already prevents AI from touching anything outside project folder
- Time Machine snapshots provide recovery for delete_file tool calls
- The real protection is containment (stay inside project), not preventing deletion

### Known gap
- When AI deletes via run_command (shell commands like rmdir) instead of delete_file tool, Time Machine does NOT capture a snapshot
- This is an edge case   future fix would be to snapshot before any destructive-looking run_command
- For now, acceptable risk since path traversal still prevents escaping project folder

### Checklist
- [x] Evaluated Option A (block directory deletion)   implemented then reverted
- [x] Decision: not needed   path traversal + Time Machine is sufficient
- [ ] Future: consider snapshotting before destructive run_command calls

## 9. TEMPLATE INSTALL SAFETY — CRITICAL — PENDING

### Threat Model
Templates are files uploaded to Supabase Storage that get written into user project folders.
For admin-curated templates: risk is near-zero (we control the content).
For future marketplace templates (user-submitted): risk increases. Scaffold templates can contain
malicious JavaScript that runs when the user opens their project in a browser or dev server.

Templates CANNOT cause traditional malware (no executables, no install scripts).
The real risk is malicious code inside project files that activates at runtime (not install time).

### Design Principles
- Templates never execute — Omnirun's existing tools do the work
- Install process is write-only: files go into the project folder, nothing else happens
- Safety is enforced by the system, not by asking users to review code

### Hard Containment (enforced by template install engine, not user choice)

**Path jail:**
- Every file a template writes MUST land inside the project folder
- System rejects ../, absolute paths, symlinks escaping the root
- Hardcoded rule in scaffold engine — no template can override it

**No outbound network during install:**
- Zero network calls during template install
- Files come from Supabase bucket and that's it
- No fetching scripts from external URLs, no phoning home

**No shell execution during install:**
- run_command is NOT available in the template install context
- This is the most dangerous capability — simply not wired into the template engine

**Tool allowlist (not blocklist):**
- Templates can ONLY use: write_file (inside project), create_directory (inside project)
- Everything else — run_command, read_file outside project, delete_file, connections — does not exist in template context
- A malicious template author can't use what doesn't exist

### Automated Scanner (runs on every upload, before template is visible)

What it checks:
- [ ] External network calls: fetch(), XMLHttpRequest, sendBeacon(), WebSocket to external domains
- [ ] External script loading: <script src="..."> pointing to non-trusted CDNs
- [ ] Data exfiltration: code accessing localStorage/document.cookie/navigator and sending externally
- [ ] Obfuscation: base64-encoded strings, eval(), Function() constructor, atob() used suspiciously
- [ ] Hidden elements: invisible iframes, zero-size forms, redirect chains
- [ ] File type validation: reject binary files disguised as text
- [ ] Size limits: max 500KB per file, max 50 files per template, max 5MB total

### Tiered Review (for future marketplace)
- Free scaffold (files only): automated scan + community flagging
- Free automation (uses tools): automated scan + tool audit + community flagging
- Paid (any type): FULL manual review before listing
- Uses run_command: ALWAYS manual review, any price

### Scheduled Task Guardrails (future automation templates)
- [ ] Can only operate on the project they were installed into
- [ ] Minimum interval: 1 hour
- [ ] Inherit same tool allowlist — no shell commands
- [ ] Maximum 5 scheduled tasks per template
- [ ] User can see/disable all template-created tasks from settings

### Kill Switch
- [ ] Any user can report a template with one click
- [ ] 3 reports → template auto-delisted, pending review
- [ ] Admin can remotely disable any template instantly
- [ ] For scheduled tasks: remote disable kills task on users' machines at next sync

### Creator Accountability (future marketplace)
- Revenue held for 7 days before payout (dispute window)
- 3 valid reports → template delisted, creator warned
- Repeat offenders permanently banned
- Creator identity tied to Omnirun account (real Stripe payment info)

### Checklist
- [ ] Path jail enforcement in templateService.ts scaffoldProjectFromTemplate()
- [ ] Validate all file paths are relative and inside project folder before writing
- [ ] Reject files with ../ or absolute paths
- [ ] Build scanner utility function (scanTemplateFiles)
- [ ] Scanner: check for external fetch/XHR/WebSocket patterns
- [ ] Scanner: check for external script tags
- [ ] Scanner: check for eval/Function/atob obfuscation
- [ ] Scanner: check for hidden iframes/forms
- [ ] Scanner: enforce file size limits
- [ ] Scanner: reject binary files

- [ ] One-click report UI in template gallery
- [ ] Auto-delist after 3 reports
- [ ] Admin kill switch in admin dashboard

### Implementation Notes
- Scanner is a utility function, not a separate service. It reads file contents and returns pass/fail with reasons.
- For Phase 1 (admin-curated templates): scanner is optional since we control the content. Build it anyway for Phase 2 readiness.
- Path jail is the most critical protection — implement first.

## 10. DESKTOP APP CONTROL SAFETY -- CRITICAL -- PARTIALLY DONE (April 3, 2026)

### See desktop-app-control.txt for full feature spec

### Threat Model
Desktop app control takes screenshots and simulates mouse/keyboard input. This is powerful and dangerous -- it could click anything, type anything, interact with any app on the user's computer.

Unlike browser control (sandboxed to Playwright), desktop control operates at the OS level. A misclick could send an email, delete a file, or make a purchase.

### What's Built

**Blocked apps list:**
- [x] User can define apps that desktop control should never interact with
- [x] Banking apps + password managers auto-suggested for blocklist ("Add common defaults" button)
- [x] Omnirun itself (omnirun, msedgewebview2) included in defaults to prevent self-control
- [x] isBlockedApp() check runs before every screenshot and action execution
- [x] Stored in localStorage via screenControlService settings

**Action delay:**
- [x] Configurable delay between actions (200ms / 500ms / 1s / 2s, default 500ms)
- [x] Gives user time to see what's happening and intervene

**Rate limiting:**
- [x] screen_capture: max 30 per minute
- [x] screen_action: max 50 per minute
- [x] Max 12 steps per screen control session (safety cap)

**Screenshot privacy:**
- [x] Screenshots sent only to Claude API (same privacy as any API call)
- [x] Screenshots NOT stored locally or remotely after processing
- [x] Base64 data exists only in memory during the loop iteration

**Stop button:**
- [x] Prominent red Stop button in floating overlay (always visible during control)
- [x] Stop button in Screen ON toggle in chat header
- [x] Stops immediately via screenControlStopRef checked every loop iteration

**AI safety instructions:**
- [x] SCREEN_CONTROL_SYSTEM_PROMPT includes "NEVER click send/submit/post, enter passwords, or interact with payment screens without user approval"
- [x] "If something unexpected appears, stop and ask the user"

### What's NOT Built Yet

**Kill switch (global hotkey):**
- [ ] Global hotkey (default F10) immediately stops all input simulation
- [ ] Hotkey works even when Omnirun is not focused (Tauri GlobalShortcutManager)
- [ ] Currently UI-only (Stop button requires Omnirun to be visible)
- CRITICAL: This is the hard safety net. Must be implemented before public release.

**Approval flow for sensitive screens:**
- [ ] Code-level enforcement: detect password/payment/send in AI response and pause
- [ ] Approval modal: "I'm about to click 'Send' on this email. Proceed?"
- [ ] User configurable: which action types always require approval
- [ ] Currently AI-prompt-only (tells AI to flag, but no code enforcement)

**Sensitive app auto-detection:**
- [ ] If AI sees banking, medical, or password fields, automatically pause
- [ ] Currently relies on blocked apps list (manual, not auto-detected)

**Session timeout:**
- [ ] Timeout after 5 minutes of no progress
- [ ] Currently only max step limit (12 steps)

### Implementation Notes
- Desktop app control safety is partially built. Blocked apps + rate limiting + action delay + stop button provide baseline safety.
- The kill switch (global hotkey) is the single most critical missing safety feature.
- Approval detection is AI-based (Claude reads the screenshot and flags sensitive content). Not 100% reliable. The kill switch is the hard safety net.
- All desktop control safety settings live in Settings > Screen Control (same page as feature toggle).
